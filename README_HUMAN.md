# 人間が書いた README
ReplayCenter は実装の全てが生成 AI によって書かれたアプリケーションであり、デプロイフローやドキュメントなど実装以外の部分も AGENTS.md 含めほぼ全て生成 AI によって書かれています。その中で、唯一人間によって全てが書かれたドキュメントとして、このファイルを作成しています。

## 開発経緯
[README](./README.md) に記載されている通り、アプリ名は NPB のリプレーセンターから取っていますが、この公開されたリプレーセンターが MLB や KBO の同種のものに比べ**あまりにもショボく見えた**ため、普段 DTV 環境を構築して野球中継を視聴している身として「これなら生成 AI だけでソフトウェア実装できるのでは？」と思ったのが発端です。

とはいえ本当にリプレー検証がしたいわけではなかったので、NPB 全試合を同時視聴できるアプリケーションとして構築することにしました。なので実際にやりたいのは工藤公康元監督の[これ](https://livedoor.blogimg.jp/livejupiter2/imgs/6/e/6e053781.jpg)です。

## コード品質
人間(b00t0x)はまったく確認していません。これまで Swift を書いたことも無ければ読んだこともなく、**実装は一瞥もしていない**ためリリース時点でも引き続き読んだことの無い状態が継続中です。テストが無いことは知ってる。

コード品質は LLM の能力と私が生成 AI にどれだけ的確な指示を与えられたかに依存しており、ネイティブ GUI アプリケーションの開発経験がゼロの人間であることを考えると後者にはあまり期待できません。一方、Web 系 IT 企業でバックエンドアプリケーションの開発を担ってきたエンジニア出身の人間ではあるので、内部実装はともかく全体アーキテクチャに関しては非エンジニアが作ったアプリケーションよりは妥当な構成になっているかもしれません。

## Q&A
### なんで macOS 専用なの？
私が macOS 使いのためです。Electron 等でクロスプラットフォーム対応することも検討しましたが、macOS 以外で利用する予定が無いため SwiftVLC が使えそうな目処が立った時点で要件から落としています。

### なんで Sequoia 以降でしか動かないの？
[SwiftVLC](https://github.com/harflabs/SwiftVLC) の要件によるものです。2026 年時点での自分のメイン環境が Sequoia のため、だいぶ厳しい条件ですが受容として SwiftVLC を採用しました。

### なんで動作環境の厳しい SwiftVLC を採用したの？
Swift ネイティブな libVLC ラッパーとして提供されており、一番実装を薄く作れそうだったためです。実装を生成 AI に丸投げしている以上、実装が複雑化して収拾つかなくなったときに対処できないため、薄く作ることを重視しています。

### EPGStation 不要にして
当初トランスコード配信メインで利用することを想定し、Mirakurun ではなくトランスコード機能のある EPGStation をバックエンドとする構成にしています。ただ、そもそも 6 ~ 9 ストリーム同時エンコードできる環境を用意する方がキツいという問題と、有線 LAN 環境での利用なので raw ts を流してクライアント側インタレ解除でも困らないということがわかり少なくとも自分は raw ts 視聴になっているので Mirakurun 対応は考え中です。

EDCB については利用していないため今のところ対応は考えていません。

### 機能追加 / バグ修正して
自分のためのアプリなので、自分も欲しいと思った場合のみ対応します。

### .xcodeproj はどこ？
無いです。先述の通りフル生成 AI 開発なので Xcode を使ってないうえ、.app バンドル化すら手組みでこうなりました。

### AGENTS.md 薄くない？
開発リポジトリの外側に decisions.md / requirements.md をはじめとした生成 AI 用の永続化ナレッジを蓄積するリポジトリを用意して開発しています。そのため、正直なところ公開リポジトリの資材のみでは開発方針を維持した改修はできないんじゃないかと思います。
